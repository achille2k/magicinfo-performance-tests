1. Tổng Quan Hệ Thống

1.1. Mục Đích

Bộ công cụ MagicInfo Performance Test Suite được thiết kế để kiểm thử hiệu năng của hệ thống quản lý màn hình Samsung MagicInfo Server. Công cụ giả lập 200 màn hình hiển thị kỹ thuật số kết nối đồng thời đến server, thực hiện đầy đủ các tác vụ mà một màn hình thực sự thực hiện trong môi trường vận hành.

ℹ️ Đối tượng sử dụng: Kỹ sư hạ tầng, quản trị viên hệ thống, nhân viên QA/QC thực hiện đánh giá năng lực chịu tải của MagicInfo Server trước khi triển khai.

1.2. Kiến Trúc Công Cụ

1.3. Kịch Bản Giả Lập

Mỗi màn hình ảo (VU — Virtual User) thực hiện đúng vòng lặp của màn hình MagicInfo thực tế:

1
Xác thực — Authentication
Màn hình gửi tên đăng nhập & mật khẩu đến server để lấy token API (POST /auth). Token được tái sử dụng trong suốt phiên kiểm thử.
2
Đăng ký thiết bị — Device Registration
Màn hình truy vấn danh sách thiết bị đã đăng ký (GET /rms/devices) để xác nhận kết nối thành công với hệ thống.
3
Heartbeat định kỳ (mỗi 30 giây)
Màn hình gửi tín hiệu "còn sống" (GET /ems/dashboard/devices/status) để server biết thiết bị đang hoạt động bình thường.
4
Kiểm tra & tải nội dung (mỗi 5 phút)

Màn hình thực hiện 3 bước liên tiếp:

  • Lấy danh sách playlist được gán (GET /cms/playlists)
  • Lấy chi tiết playlist để biết nội dung bên trong (GET /cms/playlists/{id})
  • Tải metadata của từng nội dung cần phát (GET /cms/contents/{id})

2.2. Hướng Dẫn Cài Đặt

Bước 1 — Cài đặt K6

Windows:

winget install k6 --source winget
# Hoặc tải từ: https://k6.io/docs/get-started/installation/

Linux (Ubuntu/Debian):

sudo gpg -k
sudo gpg --no-default-keyring --keyring /usr/share/keyrings/k6-archive-keyring.gpg \
     --keyserver hkp://keyserver.ubuntu.com:80 --recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D69
echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main" | \
     sudo tee /etc/apt/sources.list.d/k6.list
sudo apt-get update && sudo apt-get install k6

Bước 2 — Lấy mã nguồn

# Clone repository
git clone <repository-url> magicinfo-performance-tests
cd magicinfo-performance-tests

Bước 3 — Cài đặt Node.js dependencies

npm install

2.3. Cấu Hình Kết Nối

⚠️ Quan trọng: Không lưu mật khẩu trực tiếp vào file cấu hình. Luôn truyền qua biến môi trường hoặc file .env (không commit lên Git).

Tạo file .env từ mẫu

cp .env.example .env

Chỉnh sửa file .env:

# URL của MagicInfo Server (thay bằng địa chỉ thực tế)
MAGICINFO_BASE_URL=http://<địa-chỉ-server>:7001

# Tài khoản quản trị MagicInfo
MAGICINFO_USERNAME=admin
MAGICINFO_PASSWORD=<mật-khẩu-của-bạn>

# Số màn hình giả lập (mặc định: 200)
VIRTUAL_USERS=200

3. Hướng Dẫn Chạy Kiểm Thử

3.1. Các Kịch Bản Kiểm Thử

Kịch bảnSố màn hìnhThời gianMục đích
smoke 5 màn hình ~2 phút Kiểm tra nhanh hệ thống có hoạt động không, phát hiện lỗi cơ bản
load 200 màn hình ~7 phút Kiểm tra tải tiêu chuẩn — đánh giá hiệu năng trong điều kiện vận hành bình thường
stress 50 → 250 màn hình ~10 phút Tăng dần tải để tìm điểm giới hạn của hệ thống
soak 200 màn hình 30 phút Kiểm tra bền vững — phát hiện rò rỉ bộ nhớ và lỗi xuất hiện theo thời gian
💡 Khuyến nghị: Luôn chạy smoke trước để xác nhận kết nối thành công, sau đó mới chạy load hoặc stress.

3.2. Lệnh Chạy

Cách 1 — Dùng npm scripts (đơn giản nhất)

# Smoke Test (5 màn hình, ~2 phút) — chạy trước để kiểm tra kết nối
npm run test:smoke -- --env MAGICINFO_PASSWORD=<mật-khẩu>

# Load Test (200 màn hình, ~7 phút) — kiểm tra tải chính
npm run test:load -- --env MAGICINFO_PASSWORD=<mật-khẩu>

# Stress Test — tìm giới hạn hệ thống
npm run test:stress -- --env MAGICINFO_PASSWORD=<mật-khẩu>

# Soak Test (30 phút) — kiểm tra bền vững
npm run test:soak -- --env MAGICINFO_PASSWORD=<mật-khẩu>

Cách 2 — Lệnh K6 trực tiếp (tùy chỉnh đầy đủ)

k6 run \
  --env ENV=load \
  --env MAGICINFO_BASE_URL=http://<địa-chỉ-server>:7001 \
  --env MAGICINFO_USERNAME=admin \
  "--env=MAGICINFO_PASSWORD=<mật-khẩu>" \
  tests/scenarios/screen-simulation.js
⚠️ Lưu ý mật khẩu chứa ký tự đặc biệt (!, @, #...):
Dùng dấu ngoặc kép bao quanh tham số --env:
"--env=MAGICINFO_PASSWORD=Admin1243!"

4. Hướng Dẫn Xem Báo Cáo

4.1. Tạo Báo Cáo HTML

Sau khi test hoàn thành, file JSON kết quả được lưu tự động trong thư mục reports/. Chạy lệnh sau để tạo báo cáo HTML trực quan:

# Tạo báo cáo từ file JSON mới nhất
node scripts/generate-report.js

# Hoặc dùng npm script
npm run report

# Chỉ định file cụ thể
node scripts/generate-report.js reports/summary-load-2026-05-22T03-57-15.json

File HTML sẽ được tạo cùng tên với file JSON, ví dụ:
reports/summary-load-2026-05-22T03-57-15.html

✅ Mở file HTML bằng bất kỳ trình duyệt nào (Chrome, Edge, Firefox). Báo cáo không cần kết nối Internet (ngoại trừ biểu đồ Chart.js).

4.2. Đọc và Phân Tích Báo Cáo

Báo cáo HTML gồm 4 phần chính:

① Phần Tiêu Đề (Header)

  • Kết quả tổng: ✓ TẤT CẢ NGƯỠNG ĐẠT hoặc ✗ MỘT SỐ NGƯỠNG KHÔNG ĐẠT

② Thẻ KPI Chính (KPI Cards)

③ Bảng Kiểm Tra Ngưỡng (Thresholds)

Mỗi dòng là một tiêu chí đánh giá. Hiển thị:

  • ĐẠT — xanh: hệ thống đáp ứng yêu cầu
  • KHÔNG ĐẠT — đỏ: hệ thống không đáp ứng, cần điều tra

④ Biểu Đồ Phân Tích

  • Biểu đồ cột: So sánh thời gian phản hồi (p50 / p95 / max) theo loại hoạt động
  • Biểu đồ tròn: Phân bố số lượng request theo loại (xác thực, heartbeat, nội dung...)

5. Phân Tích và Đánh Giá Kết Quả

5.1. Tiêu Chí Đánh Giá

✅ Hệ thống ĐẠT yêu cầu khi tất cả 10 ngưỡng đều PASS, tỷ lệ lỗi HTTP = 0.00%, và API Availability = 100%.
⚠️ Cần điều tra khi p(95) của bất kỳ chỉ số nào vượt quá 80% ngưỡng cho phép. Ví dụ: HTTP p(95) = 1.800ms (gần ngưỡng 2.000ms) cần theo dõi thêm.
❌ Hệ thống KHÔNG ĐẠT khi có bất kỳ ngưỡng nào FAIL. Cần kiểm tra tài nguyên server (CPU, RAM, kết nối DB) và cấu hình hệ thống.

5.2. Xử Lý Sự Cố Thường Gặp

6. Tham Chiếu API Được Kiểm Thử

Các API REST của MagicInfo Server được công cụ gọi:

APIPhương thứcChức năng
/restapi/v2.0/authPOSTXác thực, lấy API token
/restapi/v2.0/rms/devicesGETLấy danh sách thiết bị đã đăng ký
/restapi/v2.0/ems/dashboard/devices/statusGETHeartbeat — trạng thái thiết bị
/restapi/v2.0/cms/playlistsGETLấy danh sách playlist được gán cho màn hình
/restapi/v2.0/cms/playlists/{playlistId}GETChi tiết playlist (bao gồm danh sách nội dung)
/restapi/v2.0/cms/contents/{contentId}GETMetadata chi tiết của nội dung cần phát
ℹ️ Tất cả API yêu cầu header api_key: <token> (JWT token lấy từ endpoint /auth). Token có hiệu lực 30 phút và được tự động làm mới trong quá trình test.
